ROS 2 Workshop / Differential Drive Robot Guide

ROS 2 Differential Drive Robot Workshop

Student Hands-On Guide and Verification Manual

Workshop Overview

Objective

In this workshop, you will:

  • Understand the structure of a ROS 2 robot description (URDF).
  • Build a complete differential drive robot model.
  • Visualize the robot in RViz2.
  • Validate the robot structure using URDF tools.
  • Simulate the robot in Gazebo.
  • Control the robot using ROS 2 topics.
Goal By the end of this exercise, you will have a fully functional differential drive robot that can be driven inside Gazebo simulation.

Prerequisites

Before starting, ensure:

  • ROS 2 Humble is installed.
  • Gazebo / Ignition Gazebo is installed.
  • Workshop files are available on your Desktop.
  • Terminal access is working properly.

Environment Setup

Open a terminal and run:

terminal
cd ~/Desktop/diff_drive_workshop source /opt/ros/humble/setup.bash

Verification

You should now be inside the workshop directory and have the ROS 2 environment loaded.

3Step 3
Complete Robot Assembly

Open:

urdf/04_robot_assembly.urdf.xml

Fill the missing placeholders below.

Placeholder 1 – Left Wheel Joint

<joint name="left_wheel_joint" type="continuous">
  <parent link="base_link"/>
  <child  link="left_wheel_link"/>
  <origin xyz="0.0 0.1125 -0.025" rpy="0.0 0.0 0.0"/>
  <axis xyz="0.0 1.0 0.0"/>
  <dynamics damping="0.1" friction="0.1"/>
</joint>

Understanding

This joint:

  • Connects the chassis to the left wheel.
  • Allows continuous rotation.
  • Rotates around the Y-axis.

Placeholder 2 – Right Wheel Joint

<joint name="right_wheel_joint" type="continuous">
  <parent link="base_link"/>
  <child  link="right_wheel_link"/>
  <origin xyz="0.0 -0.1125 -0.025" rpy="0.0 0.0 0.0"/>
  <axis xyz="0.0 1.0 0.0"/>
  <dynamics damping="0.1" friction="0.1"/>
</joint>

Understanding

This joint:

  • Connects the chassis to the right wheel.
  • Allows continuous wheel rotation.

Placeholder 3 – Caster Friction Configuration

<gazebo reference="caster_wheel_link">
  <mu1>0.0</mu1>
  <mu2>0.0</mu2>
  <kp>1000000.0</kp>
  <kd>100.0</kd>
  <minDepth>0.001</minDepth>
</gazebo>

Understanding

This configuration:

  • Minimizes caster wheel friction.
  • Improves simulation stability.
  • Prevents unwanted resistance during movement.

Placeholder 4 – Differential Drive Plugin

<gazebo>
  <plugin
      filename="ignition-gazebo-diff-drive-system"
      name="ignition::gazebo::systems::DiffDrive">
    <left_joint>left_wheel_joint</left_joint>
    <right_joint>right_wheel_joint</right_joint>
    <wheel_separation>0.225</wheel_separation>
    <wheel_radius>0.05</wheel_radius>
    <topic>/cmd_vel</topic>
    <odom_topic>/odom</odom_topic>
    <frame_id>odom</frame_id>
    <child_frame_id>base_footprint</child_frame_id>
    <odom_publish_frequency>50</odom_publish_frequency>
  </plugin>
</gazebo>

Understanding

This plugin:

  • Converts velocity commands into wheel motion.
  • Publishes odometry information.
  • Enables robot driving inside Gazebo.

Checkpoint

  • All placeholders completed
  • File saved successfully

4Step 4
Validate the URDF

Run:

terminal
check_urdf urdf/04_robot_assembly.urdf.xml

Expected Output

root Link: base_footprint has 1 child(ren)
    child(1):  base_link
        child(1):  caster_bracket_link
            child(1):  caster_wheel_link
        child(2):  left_wheel_link
        child(3):  right_wheel_link

What This Means

The robot has:

  • One root link
  • Proper wheel connections
  • Proper caster assembly
  • Valid robot hierarchy

Checkpoint

  • URDF validation passed
  • No parsing errors

Common Validation Error

Error: Two root links found

Cause: A wheel joint contains a typo in <parent> or <child>.

Fix: Verify all link names exactly match the link definitions.

6Step 6
Simulate and Drive the Robot

Terminal 1

Launch Gazebo:

terminal 1
ros2 launch launch/gazebo.launch.py

Wait until:

  • Gazebo opens.
  • Robot appears in simulation.

Terminal 2

Load ROS 2 environment:

terminal 2
source /opt/ros/humble/setup.bash

Move Forward

terminal 2
ros2 topic pub /cmd_vel geometry_msgs/msg/Twist "{linear: {x: 0.2}}" --once
Expected The robot moves forward.

Rotate

terminal 2
ros2 topic pub /cmd_vel geometry_msgs/msg/Twist "{angular: {z: 0.8}}" --once
Expected The robot rotates in place.

Stop Robot

terminal 2
ros2 topic pub /cmd_vel geometry_msgs/msg/Twist "{}" --once
Expected Robot stops.

7Step 7
Drive the Robot Using Teleop Keyboard

Instead of publishing individual velocity commands, you can control the robot interactively using the ROS 2 Teleop Keyboard node.

Terminal 1 – Launch Gazebo

Ensure the simulation is already running:

terminal 1
ros2 launch launch/gazebo.launch.py

Wait until the robot is fully loaded.

Terminal 2 – Start Teleop Keyboard

Source ROS 2 and launch the keyboard controller:

terminal 2
source /opt/ros/humble/setup.bash ros2 run teleop_twist_keyboard teleop_twist_keyboard

Keyboard Controls

Key Action
i Move Forward
, Move Backward
j Rotate Left
l Rotate Right
u Forward + Left
o Forward + Right
m Backward + Left
. Backward + Right
k Stop
q / z Increase / Decrease both speeds
w / x Increase / Decrease linear speed only
e / c Increase / Decrease angular speed only

Example Teleop Screen

After launching, you should see a screen similar to:

Reading from the keyboard and Publishing to Twist!

Moving around:
  u    i    o
  j    k    l
  m    ,    .

q/z : increase/decrease max speeds by 10%
w/x : increase/decrease only linear speed by 10%
e/c : increase/decrease only angular speed by 10%

CTRL-C to quit

Bonus Challenge – Autonomous Navigation Script

Instead of using the keyboard, create a ROS 2 Python node that publishes velocity commands to /cmd_vel.

Your goal is to:

  1. Move the robot forward.
  2. Rotate the robot when required.
  3. Navigate the robot into a selected room automatically.

Example command used by your node:

terminal
ros2 topic pub /cmd_vel geometry_msgs/msg/Twist \ "{linear: {x: 0.2}}" --once
